# ML Span Extractor for LaTeX Linting — Expert Briefing

## 1. Project Context

**LaTeX Perfectionist** is a formally verified LaTeX linter. It has 623 rules across 5 layers (lexer, expanded, AST, semantics, style), implemented in OCaml with Coq proofs of correctness. The production rule engine is deterministic and regex/pattern-based.

We are building an **ML span extractor** — a neural model that, given a window of raw LaTeX source text, predicts character-level spans annotated with rule IDs. Each span marks a region of text that violates a specific typographical rule (e.g., "straight quotes should be curly quotes", "use `\dots` instead of `...`", "use `\[...\]` instead of `$$...$$`").

### Why ML?

The deterministic OCaml engine already handles these rules. The ML model serves a different purpose: it provides a **probabilistic bound** that feeds into a Coq soundness proof. Specifically:

- The model is evaluated on a held-out dev set
- We compute `delta = 1 - F1` (the span-level error rate)
- This delta is imported as a concrete constant into `proofs/ML/SpanExtractorSound.v`
- The Coq proof then verifies: `1 - delta >= 0.94` (the F1 threshold from the project spec)

**The gate**: the model must achieve **span-level F1 >= 0.94** on the dev set, so that `delta <= 0.06` and the Coq proof compiles.

### Current status

After 4 major iterations, the best model achieves **F1 = 0.8503**. The gate requires 0.94. We are stuck and seeking expert guidance on how to close the gap.

## 2. Data Pipeline

### 2.1 Corpus

- **1,251 LaTeX documents** from 8 corpus directories (arXiv papers, manually curated batches)
- Documents range from 1 KB to 500 KB of raw LaTeX source
- Split: 999 train / 281 dev (80/20, stratified by rule)

### 2.2 Labeling

Labels are generated by **replaying the deterministic OCaml rule engine** in Python (`label_spans.py`). For each document:

1. Run the OCaml validators (via IPC) to get rule violation counts per document
2. A Python replay function for each rule re-implements the pattern matching to recover **exact byte offsets** (start, end) for each violation
3. Cross-validate recovered span counts against the OCaml counts
4. Output: JSONL with `{text, spans: [{start, end, rule_id}], bio_tags: [...]}`

The BIO tag scheme is standard: `O` (outside), `B-TYPO-XXX` (begin span for rule XXX), `I-TYPO-XXX` (inside span).

### 2.3 Windowing

Documents are too long for BERT (max 512 tokens ~ 400 chars). We extract overlapping windows:

- **Window size**: 300 characters
- **Stride**: 150 characters (50% overlap)
- Result: **~527K training windows**, **~144K dev windows**

### 2.4 Rule Filtering

Not all rules are learnable by a subword-tokenized model. We progressively identified and excluded rules that the model cannot learn:

| Catastrophic FP | 11 | TYPO-029 (0 TP, 1544 FP), TYPO-039 (0 TP, 1011 FP) | Model hallucinates these rules massively; generates thousands of false positives with zero true positives |
| Too rare | 4 | TYPO-009, TYPO-025, TYPO-051 | Fewer than 16 gold spans in dev set — insufficient signal to learn |

## 3. Model Architecture

### 3.1 Base Model

- **SciBERT** (`allenai/scibert_scivocab_uncased`) — a BERT variant pre-trained on 1.14M scientific papers from Semantic Scholar
- Chosen because our corpus is scientific LaTeX
- WordPiece tokenizer with 31K vocabulary
- Max sequence length: 128 tokens

### 3.2 Multi-Head Span Extractor

After the single-head approach plateaued, we moved to a **per-rule independent head** architecture:

```
Input text (300 chars)
    |
    v
SciBERT encoder (768-dim hidden states)
    |
    v
Single fused Linear(768, K*3)   # K=16 rules, 3 classes each (O/B/I)
    |
    v
Reshape to [batch, seq_len, K, 3]
    |
    v
K independent softmax heads → per-rule O/B/I predictions
    |
    v
Confidence-based merge: best non-O head per token wins
    |
    v
Subword-to-character projection (B→I interior fix)
    |
    v
BIO-to-spans: extract (rule_id, start_char, end_char) tuples
```

**Design rationale for multi-head**: In a single classifier with `2K+1` output classes, rare rules (e.g., TYPO-061 with 2 gold spans) are drowned out by common rules (TYPO-030 with 4,029 spans). Independent 3-class heads let each rule learn its own O/B/I decision boundary without interference.

**Implementation detail**: The K heads are **fused into a single `nn.Linear(768, K*3)`** matrix multiply, then reshaped. This avoids K separate CUDA kernel launches and keeps GPU utilization high.

### 3.3 Loss Function

**Focal loss** with per-head class weights:

- **Focal gamma = 2.0**: Down-weights easy negatives (O tokens are ~99.5% of data)
- **Per-head class weights**: Each head gets its own inverse-frequency O/B/I weights, capped at 20.0
- **Fully vectorized**: No Python loops — single `log_softmax` → `gather` → focal modulation → masked mean

### 3.4 Training Configuration

| Parameter | Value |
|-----------|-------|
| Optimizer | AdamW (BERT LR=3e-5, weight_decay=0.01) |
| Scheduler | Linear warmup (10%) + decay |
| Batch size | 32 |
| Max epochs | 100 |
| Early stopping patience | 25 epochs |
| Minimum epochs | 30 |
| Mixed precision | AMP (fp16) on A100 |
| TF32 | Enabled |

### 3.5 Subword-to-Character Projection

BERT tokenizes into subword units. To recover character-level spans:

1. Each subword token has an `offset_mapping` = (start_char, end_char)
2. The predicted tag for a subword is projected to its character span
3. **Critical B→I fix**: If a subword is tagged `B-RULE`, only the *first* character gets `B-RULE`; interior characters get `I-RULE`. Without this fix, every character in the subword gets `B-RULE`, creating spurious 1-character spans. This bug alone caused a 0.12 F1 drop.

### 3.6 Evaluation

**Span-level exact match**:
- Extract `(rule_id, start_char, end_char)` tuples from predicted BIO tags and gold BIO tags
- TP = intersection (exact match on all 3 fields)
- FP = predicted spans not in gold
- FN = gold spans not in predicted
- **Micro F1** is the gate metric

### 4.2 What Worked

1. **Removing CRF** (+0.56 F1): The CRF transition matrix collapsed — it learned to always predict O. Switching to independent CE loss with class weights was critical.
2. **Class-weighted focal loss**: O tokens are ~99.5% of data. Without class weights, the model predicts all-O. Inverse-frequency weights + focal gamma=2.0 force it to attend to rare B/I tokens.
3. **B→I projection fix** (+0.12 F1): The single most impactful fix. Before: all characters in a B-tagged subword got `B-RULE`, creating many spurious 1-char spans and breaking recall.
4. **Rule exclusion** (+0.03 F1): Removing unlearnable rules eliminates noise from the loss function and evaluation.

### 4.3 What Didn't Work

1. **More training time**: Model converged at epoch 19 out of 100 available. We increased patience from 10→25 and min_epochs from 10→30. The model still plateaued at epoch 19. The bottleneck is not optimization.
2. **Multi-head vs single-head**: Going from 1 classifier to K=16 independent heads gave negligible improvement on its own (both stuck at 0.70 before the B→I fix). The heads helped slightly after the fix.

## 5. Diagnosis: Why 0.85 Is the Ceiling

### 5.1 The Recall Problem

The model's **precision is 0.90** (good) but **recall is 0.81** (bad). It misses ~20% of gold spans. This is concentrated in the 3 highest-volume rules:

| Rule | What it detects | Recall | Missed spans |
|------|----------------|--------|-------------|
| TYPO-062 | `\\` backslash in text | 0.802 | 707 / 3580 |
| TYPO-030 | `----` four+ hyphens | 0.807 | 776 / 4029 |
| TYPO-028 | `$$` display math | 0.755 | 341 / 1394 |

These 3 rules account for **1,824 false negatives** out of 2,440 total (75%).

### 5.2 Root Cause: Subword Tokenization Destroys Character-Level Signals

These rules detect **specific character sequences**: `\\`, `----`, `$$`, `<`, `>`, `...`. But SciBERT's WordPiece tokenizer merges these into opaque subword tokens:

- `$$E=mc^2$$` might tokenize as `['$$', 'e', '=', 'mc', '##^', '2', '$$']` — the `$$` delimiter is one token among many, and the model must learn that *this particular token* signals TYPO-028. But `$$` also appears in many non-violation contexts (e.g., inside `\verb|$$|`).
- `\\textbf` tokenizes as `['\\', 'text', '##bf']` — the `\\` is a single token. But `\\` followed by a command is NOT a TYPO-062 violation (it's a LaTeX line break). The model must distinguish `\\` (line break = OK) from `\` (literal backslash = TYPO-062). This requires understanding LaTeX command semantics, which BERT was not pre-trained for.
- `----` might be tokenized as `['-', '-', '-', '-']` or `['----']` depending on context. The model must learn to recognize 4+ consecutive hyphens, but the tokenization is inconsistent.

**The fundamental issue**: These are **character-level pattern matching tasks** being solved by a **subword-level model**. The tokenizer acts as an information bottleneck that discards the very features the rules are looking for.

### 5.3 Exclusion Arithmetic Shows the Gap Is Structural

Even aggressively excluding rules doesn't reach 0.94:

| Strategy | Rules kept | F1 | Coverage |
|----------|-----------|-----|----------|
| Only rules with F1 > 0.94 | 4 (004, 013, 043, 048) | 0.969 | 742 / 12,744 (6%) |
| + TYPO-002 | 5 | 0.943 | 1,743 / 12,744 (14%) |
| + TYPO-030 | 6 | 0.908 | 5,772 / 12,744 (45%) |
| + TYPO-028 | 7 | 0.891 | 7,166 / 12,744 (56%) |
| + any more | 8+ | < 0.89 | — |

**To reach F1 >= 0.94 by exclusion alone, we can keep at most 5 rules covering 14% of gold spans.** This makes the ML component nearly useless.

### 5.4 Training Duration Is Not the Bottleneck

The model converges by epoch 19 out of 100 available. The learning curves show rapid early gains followed by a hard plateau — this is a **representation capacity** problem, not an optimization problem.

## 6. The Coq Proof Constraint

The Coq proof file (`proofs/ML/SpanExtractorSound.v`) currently has:

```coq
Definition measured_delta : R := 0.028.
Definition f1_threshold : R := 0.94.

Lemma eval_bound : (1 - measured_delta >= f1_threshold)%R.
Proof. unfold measured_delta, f1_threshold. lra. Qed.

Theorem span_extractor_sound_complete :
  forall delta : R,
    (delta <= measured_delta)%R ->
    (1 - delta >= f1_threshold)%R.
```

The `measured_delta` is updated when the model is retrained and passes the F1 gate. The `f1_threshold` of 0.94 comes from the project spec (§14.2 line 250). The Coq proof is trivial once the numbers check out — the hard part is making the model achieve the required F1.

**The delta = 0.028 currently in the proof is a placeholder** from a planned target. The actual model produces delta = 0.1497, which does not satisfy the bound.

### Not Yet Tried

| Approach | Rationale | Concern |
|----------|-----------|---------|
| **Character-level model** (ByT5, CharCNN+LSTM) | Operates on raw bytes — no tokenization loss | May need much larger model / more data |
| **Hybrid regex + BERT** | Regex finds candidates, BERT confirms/rejects | Adds complexity; regex already exists in OCaml engine |
| **Larger BERT** (BERT-large, DeBERTa) | More parameters, potentially better representation | 3x slower training; unclear if the bottleneck is model size |
| **Custom tokenizer** | Train a tokenizer that preserves LaTeX-specific characters | Loses SciBERT pre-training benefits |
| **Data augmentation** | Generate synthetic LaTeX with controlled violations | Could help rare rules, but high-volume rules already have thousands of examples |
| **Per-rule confidence thresholds** | Tune decision boundary per head to trade precision/recall | Could improve recall for rules with precision headroom, but unlikely to close a 0.09 F1 gap |
| **Ensembling** | Multiple models, vote | Expensive; may not help if all models hit the same tokenization ceiling |
| **Window size increase** | Larger context windows (512+ chars) | More padding, less dense signal; BERT max is 512 tokens |
| **Lowering the F1 threshold** | Accept delta=0.15, change Coq proof to `f1_threshold = 0.85` | Weakens the formal guarantee; needs stakeholder approval |
| **Two-model approach** | BERT for "semantic" rules + character model for "pattern" rules | Architectural complexity but targets the root cause |

## 8. Specific Questions for the Expert

1. **Is subword tokenization fundamentally incompatible with character-level pattern detection?** Our rules detect specific characters (`\\`, `$$`, `----`, `<`, `>`). SciBERT's WordPiece tokenizer merges these into opaque subword units. Is there a way to make a subword model attend to character-level features without switching to a character-level architecture?

2. **Would ByT5 or a byte-level model be appropriate here?** ByT5 operates on raw UTF-8 bytes and should have no tokenization information loss. But our sequences are short (300 chars) and the patterns are local. Is ByT5 overkill? Would a simpler character CNN + BiLSTM + CRF suffice?

3. **Is the multi-head architecture the right approach for multi-label span extraction?** We currently have K independent 3-class heads sharing a BERT backbone. An alternative would be a single sequence labeler with a richer tag set, or a span-proposal network (e.g., biaffine span classifier). What works best for this kind of multi-label character-level task?

4. **Is 527K training windows sufficient, or do we need more data?** The high-volume rules (TYPO-030: 4K+ spans, TYPO-062: 3.5K+ spans) still have 80% recall. Is this a data quantity problem or a representation problem?

5. **Is the 0.94 F1 threshold realistic for this task?** Given that the task requires character-level precision on a subword model, and the patterns are essentially regex-matchable, is it reasonable to expect a neural model to reach 0.94 span-level F1? Would an expert suggest lowering the threshold or changing the evaluation methodology?

6. **Should we abandon the neural approach entirely for these rules?** The OCaml engine already detects these patterns deterministically. The ML model exists solely to provide a probabilistic bound for the Coq proof. Would it make more sense to use the deterministic engine's output directly (F1=1.0 by construction) and adjust the proof strategy, rather than trying to make a neural model replicate what a regex already does perfectly?

## 11. Expert Feedback (Received 2026-03-22)

### Diagnosis

The expert identified this as a **problem-formulation ceiling**, not an optimization ceiling. Three root causes:

1. **WordPiece is the wrong inductive bias** for punctuation-run, escaping, and delimiter rules. Token-free models (CANINE, ByT5, Charformer) were proposed specifically because explicit tokenization is brittle or lossy for character-sensitive phenomena. The patterns `\\`, `$$`, `----`, `<>` are exactly where subword tokenization fails.

2. **Some labels depend on state outside the 300-char window.** For math-mode-sensitive rules (TYPO-052, TYPO-005, TYPO-028, TYPO-048, TYPO-001), the same local substring can flip label depending on math/text/verbatim state that may begin before the window. Part of the task is not identifiable from the model input.

3. **Dense BIO tagging is harder than necessary.** For most active rules, candidate boundaries are nearly deterministic once you know the anchor substring. Dense sequence labeling forces the model to discover anchors, classify context, and set exact boundaries all at once.

### Key Recommendations

- **Stop investing in plain SciBERT BIO tagging.** The ceiling is structural.
- **Switch to candidate classification**: deterministic anchor extraction + ML disambiguator + deterministic verifier.
- **Use byte/character-level models** for the classifier, not subword.
- **Add parser-state features** (in_math, in_verbatim) as explicit inputs.
- **Fix the Coq proof metric**: `1 - F1` is neither a false-positive rate nor a false-negative rate. Bound precision and recall separately.
- **Use a frozen test split** for the final evaluation imported into Coq. The current dev split has been tuned against repeatedly.
- **For fully deterministic rules, skip ML entirely** (F1=1.0 by construction).

### Expert's Recommended Architecture

```
Document text
  |
  +-- Deterministic candidate generator (reuse existing replay functions)
  |     |
  |     v
  +-- For unambiguous rules: accept directly (no ML needed)
  |
  +-- For ambiguous rules: rule-conditioned byte classifier
  |     Input: 128-byte context centered on candidate + parser-state features
  |     Output: P(violation | candidate, rule, context)
  |
  +-- Deterministic verifier (optional): confirm via OCaml engine
  |
  v
Final spans (with guarantees: soundness from verifier, completeness bounded empirically)
```

### Expert's Ranked Model Choices

1. **Best practical path**: Deterministic candidates + small byte/char classifier (CNN+BiLSTM)
2. **Best off-the-shelf token-free encoder**: CANINE or Charformer
3. **Best hypothesis test**: ByT5-small as an ablation
4. **Not recommended**: Larger BERT/DeBERTa (same tokenization bottleneck)

## 12. Chosen Architecture: Candidate Classification Pipeline

Based on the expert feedback, we are pivoting to:

### Rule Triage

**8 deterministic rules** (47% of gold spans): TYPO-002, 003, 004, 013, 030, 043, 047, 061. These are handled by existing replay functions with F1=1.0. No ML needed.

**8 ambiguous rules** (53% of gold spans): TYPO-001, 005, 012, 028, 048, 052, 056, 062. These require ML to disambiguate context (math/text/verbatim mode).

### Byte-Level Candidate Classifier

```
128-byte context window + rule_id + parser_state
  |
  +-- Byte embedding: nn.Embedding(256, 64)
  |     |
  |     v
  |   Dilated CNN (kernels=[3,5,7], dilation=[1,2,4])
  |     |
  |     v
  |   BiLSTM (hidden=128)
  |     |
  |     v
  |   Anchor-pooled representation (256-dim)
  |
  +-- Rule embedding (16-dim)
  +-- Parser features: [in_math, in_verbatim, after_command, depth] (4-dim)
  |
  v
MLP(276, 128, 1) -> sigmoid -> P(violation)
```

- **~2M parameters** (vs 110M SciBERT)
- **No tokenization loss** (raw bytes)
- **Balanced training** (~50/50 pos/neg by construction)
- **Explicit parser state** (eliminates math/verbatim ambiguity)

### Coq Proof Refactor

Replace `delta = 1 - F1` with separate precision and recall bounds, evaluated on a frozen test split.
